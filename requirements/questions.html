Developer-Client Meeting Questions Team 2

February 8, 2016 ( Last Updated: February 10th, 2016 by Jessica DiMariano )

Purpose

The client has asked our team to develop a tool for standardizing postal addresses. The questions listed on this document will be asked when interviewing the client. By identifying the users, goals, functionality, and success metrics, these questions aim to understand the needs of the client. The answers to our questions will aid our team in the creation of a requirements document.

  1. 1) Questions Related to the User:
    1. 1.a Who will be using this system?
      1. 1.a.i Do you want restrictions on who should be able to use this system? For example, would you only want a manager to be able to use this?
      2. 1.a.ii How experienced with technology do you expect a user of this system to be?
        1. 1.a.ii.1 Would they be comfortable with a command-line?
    2. 1.b Will the user be at their computer the entire time while the program is running or will the program be started and left?
  2. 2) Questions Related to Processing the Data Itself:
    1. 2.a Will this software need to interact with the internet at all?
    2. 2.b How many addresses do you want the system to process at a time?
    3. 2.c Will you be using a collection of addresses or will they be entered one at a time manually?
    4. 2.d How should the software handle an address or zip code mismatch?
    5. 2.e Does the software need to detect if a particular address does not exist?
      1. 2.e.i What do you want the system to do if it recognizes that a particular address does not exist?
    6. 2.f How much of an address should be automatically filled in if parts are missing?
    7. 2.g Do you want this system to be able to identify if an address is not actually present in the US?
    8. 2.h What do you want the system to do if it recognizes a common spelling error?
    9. 2.i How much tolerance would you have for the system making a mistake?
      1. 2.i.1 Are there any mistakes the system might make that you would consider acceptable?
    10. 2.j How fast do you need this system to be?
    11. 2.k How much configuration and customization should the software have at its disposal?
      1. 2.k.1 Should there be anything more than a “Run Program”-type button, or should it handle custom editing fields such as “Capitalize First Letters” instead of “ALL CAPS” as per USPS standards?
      2. 2.k.2 Do you want a user or a manager of the system to be able to make any modifications or do you want to leave it up to the developers themselves?
  3. 3) Questions Relating to Input and Output
    1. 3.a If this system is taking in addresses from a collection, do you want it to replace those addresses with the new corrected addresses or do you want a new collection to be started?
    2. 3.b How will the addresses be given to the software?
      1. 3.b.i If a user is entering an address, will they be entering the information into separate fields such as “zip code” or “state”?
      2. 3.b.ii If there is a collection of addresses, would the address be divided by zip code, state, etc?
  4. 4) Miscellaneous Questions
    1. 4.a How will the common user be interacting with the software?
    2. 4.b On what platform will this software be interfacing? (e.g., a desktop computer? A server command- line? If so, what kind of operating system? What server version?)
    3. 4.c What kind of system capabilities can we expect?
