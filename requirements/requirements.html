USPS Mailing Address Formalization System Requirements DocumentTeam 2

February 15, 2016 (Last Updated: April 11th, 2016 by Aaron Gershman)

Person Responsible for Document: Aaron Gershman

Purpose

This is the requirements document for a USPS Mailing Address Formalization System. The document is formed based on the questions the client was asked during an interview on February 12th, 2016 in Violette Hall 1328 at Truman State University. The client contact spoken to was Dr. Jon Beck. The USPS Mailing Address Formalization System (colloquially known as “USPSMAFS”) is a tool to convert mailing addresses into the United States Postal Services’ standards for domestic U.S. mail. This requirements outline is an evolving document, and is subject to change in accordance to the needs of the client and the system itself.

  1. 1 Success Criteria
    1. 1.1 A successful implementation of USPSMAFS will...
      1. 1.1.1 convert supplied valid addresses into the USPS standard, and
      2. 1.1.2 reject invalid addresses, making the user aware that these addresses could not be formalized.
      1.2 This metric assumes...
      1. 1.2.1 all supplied addresses are spelled correctly, and
      2. 1.2.2 all supplied addresses are complete (e.g., include a ZIP code), and
      3. 1.2.3 the system will not verify if an address actually exists (e.g., “123 Happy Street, Kirksville MO 63501-9999” will be properly formalized, but whether the address exists cannot be determined).
  2. 2 Functional Requirements
    1. 2.1 The user will have access to two different validation modes: individual address mode, and bulk address mode. (1.a.ii, 2.c, 4.a)
      1. 2.1.1 Individual address mode allows the user to enter in an address into separate fields (street address, city, state, zip-5, zip-4), and the system will give back a formalized address. (3.b)
      2. 2.1.2 Bulk address mode allows the user to load a source file of addresses, run the program, and two output files will be generated: one of formalized addresses, and the other of addresses which could not be validated. (3.b)
  3. 3 Functional Requirements: Scenarios
    1. 3.1 User “Moses” starts the USPSMAFS on their computer, and is presented with a mainscreen showing two mode options. Moses would like to formalize a single address, so they select the “individual address mode.” (4.a, 2.c)
      1. 3.1.1 Moses enters the address into separate address fields (e.g., street address, city, state, zip-5, zip-4). (3.b.i)
      2. 3.1.2 The formalized address is then displayed on screen alongside the address originally entered. (3.b.i)
    2. 3.2 User “Jesus of Nazareth” would like to formalize their address database in bulk. Jesus opens the USPSMAFS, and selects “bulk address mode.” (4.a)
      1. 3.2.1 Jesus clicks the “input file source” button, and navigates to the input file location. (4.a)
      2. 3.2.2 Jesus clicks the “output file destination” button, and chooses where they would like the output file to be. (4.a)
      3. 3.2.3 Jesus then clicks the “run USPSMAFS” button, upon which the system begins the formalization process. A status bar is displayed to inform Jesus that the system is running. (4.a)
      4. 3.2.4 When the system is finished, Jesus is presented with a “Summary Report” prompt. Jesus is told that 6,920 addresses were properly formalized and saved to output, but 40 addresses were invalid, and saved to a different output. (4.a)
  4. 4 Nonfunctional Requirements: Bulk Address Mode
    1. 4.1 The bulk address mode should allow the user to specify both the input and output file destinations. (3.a)
    2. 4.2 The bulk address mode will not overwrite or modify the input data file in any way. All modifications made to the data are to be saved in a different output. (3.a)
    3. 4.3 The order of the output file should be preserved as from the input file. (4.a)
    4. 4.4 The presumed input file type will be a CSV file. (3.b)
    5. 4.5 The presumed output file type will be a CSV file. (3.b)
    6. 4.6 Must be able to handle, at least, up to 50,000 entries. (2.b)
  5. 5 Nonfunctional Requirements: General
    1. 5.1 Must be portable to linux/mac/windows. (4.b, 4.c)
    2. 5.2 Must be accessible as a standalone program on standard laptop/desktop environments. (4.b, 4.c)
    3. 5.3 Should be able to handle the different kinds of addresses as described in P28 of the USPS formatting manual. Specifically, BUSINESS and HOME addresses. No names on the addresses will be included. (2.k.2)
    4. 5.4 The scope of the system pertains to U.S. Addresses. Standardizing to outside nations such as Canada or Mexico are outside the scope of this program. (2.g)
    5. 5.5 The standard user interface will be through a GUI. (4.a, 1.a.ii, 1.a.iii)
  6. 6 Nongoals of USPSMAFS
    1. 6.1 No time or space requirements. (2.j)
    2. 6.2 No need to sort the data. (3.a)
    3. 6.3 No option to customize formatting options (e.g., the system only needs to adhere to USPS P28 standards). (2.k.1, 2.k.2)
    4. 6.4 No need to fix spelling mistakes. (2.i, 2.h)
    5. 6.5 No need to attempt to fill missing data (e.g., attempting to fill in missing ZIP information). (2.d)
    6. 6.6 No need to catch duplicate entries. (2.d)
    7. 6.7 No need to confirm if an address actually exists or not. (2.g, 2.d)
    8. 6.8 No need for user security authentication. (2.a, 4.a, 4.b)
  7. 7 Glossary, definitions, abbreviations, or acronyms as needed:
    1. 7.1 USPSMAFS: United States Postal Service Mailing Address Formalization System
    2. 7.2 Complete address: all address fields (street address, city, state, zip-4, zip-5) are filled in with data.
    3. 7.3 Valid address: an address which is complete, and the data can be properly formatted (e.g., the “State” field contains “Missouri,” which can be formalized into “MO”)
    4. 7.4 Formalized address: an address which has been successfully converted into the USPS P28 mailing standard.
